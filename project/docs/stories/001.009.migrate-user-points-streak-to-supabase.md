## Problem Statement

**CRITICAL DATA INTEGRITY ISSUE** 🔴

Currently, user points and streak are stored as:
- ❌ **PRIMARY storage:** localStorage (`user_points`, `user_streak`)
- ⚠️ **SECONDARY storage:** Supabase `profiles.total_points` (updated asynchronously)
- ❌ Streak stored ONLY in localStorage (zero persistence on logout/device change)
- ❌ App uninstall = complete data loss (points, streak, progress)
- ❌ Device change = points appear 0 (must wait for async sync)
- ❌ localStorage clear = all data lost

This causes:
- 🔴 **Data loss on logout/uninstall** (user loses ALL points + streak)
- 🔴 **Multi-device inconsistency** (different points on phone vs web)
- 🔴 **Rankings based on stale data** (calculateUserRank() reads localStorage)
- 🔴 **Streak reset risk** (stored only in localStorage, not Supabase)
- 🔴 **Zero audit trail** (no history of points earned/lost)
- 🔴 **Production blocker** (current architecture cannot handle production use)

**This story fixes the foundation: Make Supabase the PRIMARY storage for points and streak.**

- [ ] `user_point_history` table created with:

## Risks & Blockers

### Blocker: CRITICAL - Data Loss Risk (CRITICAL - MUST MITIGATE)
- **Issue:** localStorage is PRIMARY storage now; points lost on logout/uninstall
- **Mitigation:** Task 1.9.4 MUST migrate ALL historical points before enabling Supabase
- **Impact:** If not careful, users lose their points permanently
- **Ownership:** @data-engineer reviews migration script line-by-line; @dev validates 100% success

### Blocker: Real-time Listeners Reliability (High)
- **Issue:** If connection drops, points updates silently fail
- **Mitigation:** Implement fallback polling (30s); retry logic with exponential backoff
- **Impact:** User sees stale points until page refresh
- **Ownership:** @dev implements robust connection handling

### Risk 1: Timezone Issues in Streak Calculation (Medium)
- **Issue:** Streak reset logic depends on "today's date" but timezones vary globally
- **Mitigation:** Use UTC for all dates in Supabase; calculate user's local date in app
- **Impact:** Users in different timezones may see incorrect streak values
- **Ownership:** @dev documents timezone handling clearly

### Risk 2: Race Conditions in Point Updates (Medium)
- **Issue:** Multiple rapid updates (mobile + web simultaneously) could cause duplicates
- **Mitigation:** Use database transactions; implement idempotency via reference_id
- **Impact:** Points could be counted twice or not at all
- **Ownership:** @dev uses transactions in addUserPoints(); tests with concurrent updates

### Risk 3: Migration Performance (Medium)
- **Issue:** Migrating points for 1000+ users may be slow
- **Mitigation:** Batch inserts in sets of 100; monitor performance
- **Impact:** Migration may take minutes; need to plan downtime or batch process
- **Ownership:** @dev monitors migration speed; can be run in background
