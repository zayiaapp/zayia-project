## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Real-time listeners cause memory leaks | HIGH | Use cleanup on unmount, test with DevTools |
| Race conditions in rankings calculation | MEDIUM | Use database trigger + LOCK on monthly calculation |
| Message flood (spam attack) | MEDIUM | Implement rate limiting at API level |
| Ban system circumvention | MEDIUM | Check ban status before allowing message POST |
| Concurrent reactions cause duplicates | LOW | Use UNIQUE constraint (user_id, message_id, emoji) |

## Dev Notes

**Real-Time Architecture:**
- Use Supabase `postgres_changes` (realtime subscription)
- Subscribe to specific tables on component mount
- Unsubscribe on component unmount
- Use `useEffect` cleanup function for proper cleanup
- Handle connection loss gracefully (show "Offline" indicator)

**Performance Considerations:**
- Paginate messages (load 20 per scroll)
- Debounce rapid updates (e.g., typing indicator)
- Use `useCallback` for listener handlers
- Lazy load message avatars
- Cache monthly rankings for 1 day

**Security Notes:**
- RLS policies enforce user can only insert own messages
- Ban check happens server-side (RLS policy)
- Soft delete ensures audit trail
- Reports go to admin review (email alert in Phase 4)
- All moderation actions logged

**Testing Strategy:**
1. Single tab: post message, verify appears
2. Two tabs: post in tab A, verify in tab B instantly
3. Reactions: add emoji in tab A, verify in tab B
4. Delete: soft delete in tab A, verify appears deleted in tab B
5. Ban: ban user, verify messages greyed out
6. Rankings: complete challenge, verify ranking position updates instantly
7. Performance: post 100 rapid messages, check no lag
8. Cleanup: close tabs, verify no listener warnings in console
