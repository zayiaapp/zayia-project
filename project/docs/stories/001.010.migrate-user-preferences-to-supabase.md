## Problem Statement

Currently, user preferences are stored as:
- ❌ localStorage (scattered across multiple keys)
- ❌ No database record of preferences
- ❌ Preferences lost on logout
- ❌ No sync across devices
- ❌ No admin visibility into user preferences
- ❌ Can't analyze preference trends

Preferences include:
- Daily challenge goal (5, 10, 15 challenges per day)
- Notification settings (enabled/disabled, quiet hours)
- Theme preference (light/dark)
- Language preference (Portuguese/English)
- Email frequency (daily/weekly/never)

**This story creates Supabase tables for user preferences with full sync.**

## Acceptance Criteria

### AC1: Create Supabase Tables
- [ ] `user_preferences` table created with:
  - `id` (UUID, primary key)
  - `user_id` (UUID, unique, foreign key → auth.users)
  - `daily_goal` (integer, default 5, range 1-20)
  - `notifications_enabled` (boolean, default true)
  - `quiet_hours_start` (time, nullable, e.g., "22:00")
  - `quiet_hours_end` (time, nullable, e.g., "07:00")
  - `theme` (text, enum: light/dark/auto, default "auto")
  - `language` (text, enum: pt/en, default "pt")
  - `email_frequency` (text, enum: daily/weekly/never, default "weekly")
  - `marketing_emails` (boolean, default true)
  - `created_at`, `updated_at` (timestamps)
- [ ] Proper indexes on user_id and created_at
- [ ] Check constraints on enums and ranges

### AC2: Seed from localStorage
- [ ] Create migration: `supabase/migrations/20260304000012_seed_user_preferences.sql`
- [ ] For each user:
  - Read localStorage daily goal
  - Read localStorage notifications settings
  - Read localStorage theme preference
  - Insert to user_preferences table
- [ ] Seed data is idempotent (safe to run multiple times)
- [ ] Users without preferences get defaults

### AC3: Create Supabase Client Functions
- [ ] Add `getUserPreferences(userId)` function:
  - Returns complete user preferences object
  - Cached with 5-minute TTL
  - Creates default preferences if missing
- [ ] Add `updateUserPreferences(userId, updates)` function:
  - Updates one or more preferences
  - Validates enum values and ranges
  - Returns updated preferences
  - Triggers real-time update
- [ ] Add `resetUserPreferences(userId)` function:
  - Resets to defaults
  - Useful for troubleshooting

### AC4: Update Components to Use Supabase
- [ ] `ProfileSection.tsx` / `PreferencesSection.tsx` refactor:
  - Load preferences from `getUserPreferences()`
  - Update via `updateUserPreferences()`
  - Same UI/UX (no visual changes)
- [ ] `DashboardSection.tsx` update:
  - Reads `daily_goal` from Supabase, not localStorage
- [ ] `ChallengesSection.tsx` update:
  - Uses daily_goal from Supabase for progress calculation

### AC5: Real-time Synchronization
- [ ] Real-time listener in preferences UI:
  - Subscribe to `user_preferences` for current user
  - Auto-update when preference changes
  - Enables multi-tab sync (change in one tab = update in all)
- [ ] Listener cleans up on unmount
- [ ] Fallback to polling (60s) if connection drops

### AC6: RLS Policies
- [ ] `user_preferences`: Users see/edit own preferences
  - SELECT: `auth.uid() = user_id`
  - INSERT: `auth.uid() = user_id` (auto-creates on first update)
  - UPDATE: `auth.uid() = user_id`
  - DELETE: Disabled (preferences always exist)
- [ ] Admins can read all user preferences (for analytics)

### AC7: Quality Gates
- [ ] `npm run build` succeeds with 0 TypeScript errors
- [ ] `npm run lint` passes with 0 warnings
- [ ] Migration idempotent (runs 2+ times safely)
- [ ] Manual test: Update preference → persisted to Supabase
- [ ] Manual test: Logout/login → preferences still exist
- [ ] Manual test: Change in one tab → updates in other tab (real-time)

## Risks & Blockers

### Risk 1: localStorage Keys Unknown (Medium)
- **Issue:** Don't know exact localStorage key names used for preferences
- **Mitigation:** During Task 1.10.1, inspect localStorage to find preference keys
- **Impact:** If keys don't match expectations, migration fails; need to investigate
- **Ownership:** @dev checks localStorage during kickoff

### Risk 2: Enum Validation Strictness (Low)
- **Issue:** If user has invalid preference value in localStorage, migration fails
- **Mitigation:** Implement validation with defaults; skip invalid entries with logging
- **Impact:** Some users' preferences not migrated; fallback to defaults
- **Ownership:** @dev implements forgiving migration logic

### Risk 3: Multi-tab Sync Latency (Low)
- **Issue:** Real-time updates may have slight delay between tabs
- **Mitigation:** Acceptable; document that users may see old preference for 1-2 seconds
- **Impact:** Minor UX issue; not blocking
- **Ownership:** @qa notes as known limitation

### Risk 4: Quiet Hours Timezone (Low)
- **Issue:** quiet_hours_start/end are times, not timestamps; timezone handling unclear
- **Mitigation:** Store as UTC times; calculate user's local quiet hours in app
- **Impact:** Notifications may fire at wrong time for users in different timezones
- **Ownership:** @dev documents timezone approach clearly
