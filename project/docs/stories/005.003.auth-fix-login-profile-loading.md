## Problem Statement

### Problema 1: Full Table Scan no Login (Performance Crítica)

Arquivo: `src/contexts/AuthContext.tsx`, linhas 117-120

```typescript
// CÓDIGO ATUAL — ERRADO
const profiles = await supabaseClient.getProfiles() // SELECT * FROM profiles (TODOS)
const userProfile = profiles.find((p: SupabaseProfile) => p.id === session.user.id) // filtra em JS
```

**Por que é um problema:**
- `getProfiles()` faz `SELECT * FROM profiles` sem WHERE → traz TODOS os usuários
- Com 100 usuárias: 100 rows transferidas para encontrar 1
- Com 1000 usuárias: 1000 rows para encontrar 1
- RLS limita o que o usuário vê, mas a query ainda é ineficiente
- Cada login faz um full scan desnecessário

### Problema 2: Profile Type Duplicado (Bug Silencioso)

Existem **dois** tipos `Profile` divergentes:
- `src/lib/supabase-client.ts` — tipo do banco (snake_case flat, ~25 campos)
- `src/contexts/AuthContext.tsx` linha 6 — tipo local com `address` como objeto aninhado

O AuthContext importa `SupabaseProfile` do supabase-client mas define o próprio `Profile` localmente. Os dois coexistem e causam confusão sobre qual usar.

- [ ] Adicionar em `src/lib/supabase-client.ts`:

```typescript
// Busca UM perfil por ID — use este, nunca getProfiles() para usuário logado
async getProfile(userId: string): Promise<Profile | null> {
  try {
    const { data, error } = await supabase
      .from('profiles')
      .select('*')
      .eq('id', userId)
      .single() // Retorna objeto direto, não array. Lança erro se 0 ou > 1 resultado.
    if (error) throw error
    return data
  } catch (err) {
    console.error('❌ getProfile error:', err)
    return null
  }
}
```

## Risks & Blockers

### Risk 1: `address` object em ProfileSection.tsx (Médio)
- **Problema:** `ProfileSection.tsx` pode estar construindo `address: { street, city, ... }` para passar ao `updateProfile()`. Com tipo unificado, o correto é passar `street`, `city` diretamente (flat).
- **Mitigação:** Buscar em `ProfileSection.tsx` por `address:` e corrigir.
- **Como detectar:** `grep -n "address" src/components/user/sections/ProfileSection.tsx`
- **Impacto:** Se não corrigir, TypeScript lança erro de tipo e build falha.

### Risk 2: `SupabaseProfile` alias confunde o dev (Baixo)
- **Problema:** O AuthContext importava como `type Profile as SupabaseProfile` para evitar conflito com o Profile local. Após remover o Profile local, o alias pode ficar.
- **Mitigação:** Remover o alias e usar `Profile` direto do supabase-client.
- **Impacto:** TypeScript warning/error fácil de corrigir.

### Risk 3: `getProfiles()` usado em outros components (Baixo)
- **Problema:** GuerreirasSection ou outro componente admin pode estar usando `getProfiles()` para listar todas as usuárias — isso é legítimo para admin.
- **Mitigação:** Verificar com grep antes de qualquer remoção. NÃO remover `getProfiles()` se ainda for usado.
- **Impacto:** Zero se não remover o método.
