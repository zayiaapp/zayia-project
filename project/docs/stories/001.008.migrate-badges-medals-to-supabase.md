# STORY 001.008: Migrate Badges & Medals to Supabase

**Status:** Ready for Review ✅

## Problem Statement

Currently, badges/medals are stored as:
- ❌ Hardcoded constant `BADGES` array in `/src/lib/badges-data-mock.ts` (50+ items)
- ❌ Hardcoded `LEVELS` constant in same file (10 levels)
- ❌ No database source of truth (all logic in-memory)
- ❌ `user_earned_badges` tracked only in localStorage
- ❌ Can't track medal unlock history
- ❌ Can't add new badges/medals without code deployment

This prevents:
- Dynamic medal creation/management
- Admin interface to create custom medals
- Multi-device medal sync (localStorage is device-local only)
- Medal unlock analytics and reporting
- Cross-device badge progress visibility

**This story creates Supabase tables for badges, levels, and user medal tracking.**

## Acceptance Criteria

### AC1: Create Supabase Tables
- [ ] `badges` table created with:
  - `id` (UUID, primary key)
  - `badge_id` (text, unique identifier like "saude_iniciante")
  - `name` (text, display name)
  - `description` (text, what to earn it)
  - `category` (text, enum: category/global/level)
  - `icon` (text, emoji)
  - `points` (integer, points awarded when unlocked)
  - `requirement` (JSONB, unlock condition: {type: "total_completed", value: 1})
  - `display_order` (integer)
  - `is_active` (boolean)
  - `created_at`, `updated_at` (timestamps)
- [ ] `levels` table created with:
  - `id` (UUID, primary key)
  - `level_number` (integer, 0-9)
  - `name` (text, level name)
  - `min_points` (integer, minimum points for this level)
  - `max_points` (integer, maximum points in this level)
  - `icon` (text, emoji)
  - `created_at`, `updated_at` (timestamps)
- [ ] `user_earned_badges` table created with:
  - `id` (UUID, primary key)
  - `user_id` (UUID, foreign key → auth.users)
  - `badge_id` (UUID, foreign key → badges)
  - `earned_at` (timestamp)
  - Unique constraint: (user_id, badge_id)

### AC2: Seed Data from Mock Constants
- [ ] Create migration: `supabase/migrations/20260304000004_seed_badges.sql`
- [ ] Insert all badges from `BADGES` constant (~50 items):
  - Category badges (saude_iniciante, saude_veterana, etc.)
  - Global badges (Ovo, Iniciante, etc.)
  - Level badges (if any)
- [ ] Insert all levels from `LEVELS` constant (Bronze → Diamond)
- [ ] Seed data is **idempotent** (safe to run multiple times)
- [ ] All existing user_earned_badges from localStorage migrated to Supabase

### AC3: Create Supabase Client Functions
- [ ] Add `getAllBadges()` function:
  - Returns all active badges
  - Cached with 5-minute TTL
- [ ] Add `getBadgesByCategory(category)` function:
  - Returns badges filtered by category
- [ ] Add `getUserEarnedBadges(userId)` function:
  - Returns badges earned by specific user
  - Includes earned_at timestamp
- [ ] Add `checkAndAwardBadge(userId, badgeId)` function:
  - Awards badge to user if not already earned
  - Returns success/already-owned
  - Validates requirement met (total_completed, points, etc.)
- [ ] Add `getAllLevels()` function:
  - Returns all levels sorted by level_number
- [ ] Add `getUserLevel(userId)` function:
  - Calculates current level from total_points
  - Returns level object

### AC4: Migrate localStorage Earned Badges to Supabase
- [ ] Create data migration script:
  - For each user in auth.users
  - Read localStorage earned badges
  - Insert to user_earned_badges table
  - Log migration progress
- [ ] All historical earned_at timestamps preserved if available
- [ ] After migration, localStorage badges remain (for fallback) but Supabase is primary

### AC5: Update Components to Use Supabase
- [ ] `BadgesSection.tsx` / `MedalSection.tsx` refactor:

  - Replace hardcoded requirement checks with Supabase logic

- [ ] `checkAndUnlockMedals()` function refactor:
  - Takes Supabase `checkAndAwardBadge()` function call

- [ ] `PopUpMedalUnlocked.tsx`:
  - Still works (data source unchanged for UI)
- [ ] `RankTierBadge.tsx`:

  - No visual changes

### AC6: RLS Policies Implemented

## Risks & Blockers

### Risk 1: BADGES Constant Structure Mismatch (Medium)
- **Issue:** `BADGES` constant in `/src/lib/badges-data-mock.ts` may have unexpected structure
- **Mitigation:** During Task 1.8.1, review constant structure to confirm all fields map to badges table
- **Impact:** Seed migration may need adjustments, +30 min rework
- **Ownership:** @data-engineer verifies before migrations

### Risk 2: Badge Requirement Logic Complexity (Medium)
- **Issue:** Badge requirements are complex (global medals, category medals, different trigger types)
- **Mitigation:** Implement JSONB requirement column flexibly; document example requirements
- **Impact:** If structure too rigid, unlock logic breaks; need to support different trigger types
- **Ownership:** @data-engineer designs schema; @dev implements flexible requirement checking

### Blocker: localStorage Migration Data Loss (High)
- **Issue:** Migrating earned badges from localStorage → Supabase without verification could lose data
- **Dependency:** Task 1.8.3 must carefully migrate with validation checks
- **Resolution:** Validate 100% of users' earned badges migrated successfully
- **Ownership:** @dev writes careful migration script with logging

### Risk 3: Circular Dependency with Points (Medium)
- **Issue:** Badge unlocking may depend on points from STORY-001.009
- **Mitigation:** Ensure STORY-001.009 completes first (points persisted to Supabase)
- **Impact:** Cannot properly test badge unlocks if points still in localStorage
- **Ownership:** @pm sequenced STORY-001.009 before 001.008
