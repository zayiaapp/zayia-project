## Problem Statement

With new Supabase tables created (challenges, badges, user_points, preferences), we need to verify:
- ❌ All RLS policies implemented correctly
- ❌ No accidental public writes (security risk)
- ❌ Users can only access their own data (privacy)
- ❌ Admins/CEOs have elevated access (admin dashboard)
- ❌ Cross-device access patterns work (mobile → web)
- ❌ No orphaned data without RLS protection

**This story tests and documents all RLS policies.**

## Risks & Blockers

### Blocker: RLS Test Environment Setup (High)
- **Issue:** Testing RLS requires ability to login as different users, including fake test users
- **Mitigation:** Create test user accounts during Task 1.11.1 setup
- **Impact:** Cannot test cross-user policies without test accounts
- **Ownership:** @data-engineer creates test users before running test suite

### Risk 1: Overly Permissive Default Policies (High)
- **Issue:** If default RLS on tables is too permissive, data exposure risk
- **Mitigation:** Task 1.11.1 tests all policies; any failures = STOP, fix before merge
- **Impact:** Private user data (preferences, points) could be readable by others
- **Ownership:** @qa gates PR merge on 0 RLS test failures

### Risk 2: Foreign Key Orphaned Data (Medium)
- **Issue:** Without proper foreign key constraints, orphaned data could violate RLS intent
- **Mitigation:** Verify all foreign keys are present and ON DELETE CASCADE/SET NULL configured
- **Impact:** Deleted user leaves orphaned data accessible via bypass
- **Ownership:** @data-engineer confirms constraints during schema audit

### Risk 3: Service Role Key Misuse (Medium)
- **Issue:** Service role bypasses RLS; if exposed, data breach
- **Mitigation:** Document explicitly where service role is used; restrict access
- **Impact:** Credentials leaked = all RLS policies bypassed
- **Ownership:** @dev documents all service role usage; @devops restricts environment access

### Risk 4: RLS Policy Update Vulnerability (Low)
- **Issue:** RLS policies can be updated accidentally via schema migrations
- **Mitigation:** Review all RLS-related migrations carefully; test after updates
- **Impact:** Policy changes silently break data isolation
- **Ownership:** @data-engineer reviews migrations; @qa verifies after each update

## Testing Scenarios

### Anonymous User (not logged in)
```sql
-- Can read public data
SELECT * FROM challenges; ✅

-- Cannot read private data
SELECT * FROM user_preferences; ❌ (RLS denial)
```

### Authenticated User (User A)
```sql
-- Can read own preferences
SELECT * FROM user_preferences WHERE user_id = auth.uid(); ✅

-- Cannot read other user's preferences
SELECT * FROM user_preferences WHERE user_id != auth.uid(); ❌ (RLS denial)
```

### CEO/Admin User
```sql
-- Can read all data
SELECT * FROM user_preferences; ✅ (all rows)
SELECT * FROM user_point_history; ✅ (all rows)
```
