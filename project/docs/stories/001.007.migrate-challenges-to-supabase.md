# STORY 001.007: Migrate Challenges to Supabase

**Status:** Ready for Review ✅
**Complexity:** 3 points
**Created:** 2026-03-04
**Epic:** EPIC-001.1: Data Integration & Migration
**Depends On:** STORY-001.001 (Backend Phase 1 - Auth)

---

## Problem Statement

Currently, challenges are stored as:
- ❌ JSON files in `/src/data/*.json` (8 category files)
- ❌ Hardcoded in memory (no database source of truth)
- ❌ Updated only via code deployment (can't add challenges dynamically)
- ❌ ChallengesSection reads from JSON/localStorage combo

This prevents:
- Dynamic challenge creation without redeployment
- Admin interface to manage challenges
- Real-time challenge updates across devices
- Challenge analytics and tracking

**This story creates Supabase tables for challenges and migrates all mock data.**

## Acceptance Criteria

### AC1: Create Supabase Tables
- [ ] `challenge_categories` table created with:
  - `id` (UUID, primary key)
  - `name` (text, category name: "Saúde Corporal", "Carreira", etc.)
  - `icon` (text, emoji or icon identifier)
  - `description` (text, optional)
  - `display_order` (integer, for ordering)
  - `created_at`, `updated_at` (timestamps)
- [ ] `challenges` table created with:
  - `id` (UUID, primary key)
  - `category_id` (foreign key → challenge_categories)
  - `title` (text, challenge name)
  - `description` (text, full description)
  - `points` (integer, 5/10/15/20/25)
  - `difficulty` (enum: easy/medium/hard)
  - `display_order` (integer, within category)
  - `is_active` (boolean, for soft-delete/disable)
  - `created_at`, `updated_at` (timestamps)
- [ ] Both tables have proper indexes (category_id on challenges)

### AC2: Seed Data from JSON Files
- [ ] Create migration file: `supabase/migrations/20260304000001_seed_challenges.sql`
- [ ] Insert 8 categories from `/src/data/` files:
  - corpo_saude.json → "Saúde Corporal"
  - carreira.json → "Carreira"
  - relacionamentos.json → "Relacionamentos"
  - mindfulness.json → "Mindfulness"
  - digital_detox.json → "Digital Detox"
  - rotina.json → "Rotina"
  - compliance.json → "Compliance"
  - autoestima.json → "Autoestima"
- [ ] Insert all challenges from each category (50+ total)
- [ ] Seed data is **idempotent** (safe to run multiple times)

### AC3: Create Supabase Client Functions
- [ ] Add `getChallenges()` function:
  - Returns all active challenges with category relationship
  - Example: `const challenges = await getChallenges();`
  - Returns: `Promise<{id: UUID, title: string, points: number, category_id: UUID, difficulty: enum, ...}[]>`
  - Cached with 5-minute TTL
  - TypeScript type: `Challenge[]`
- [ ] Add `getChallengesByCategory(categoryId)` function:
  - Returns challenges filtered by category_id, sorted by display_order
  - Example: `const catChallenges = await getChallengesByCategory('uuid-here');`
  - Returns: `Promise<Challenge[]>`
- [ ] Add `getChallengeById(challengeId)` function:
  - Returns single challenge with full details
  - Example: `const challenge = await getChallengeById('uuid');`
  - Returns: `Promise<Challenge>`
- [ ] Add `getChallengeCategories()` function:
  - Returns all categories sorted by display_order
  - Example: `const categories = await getChallengeCategories();`
  - Returns: `Promise<{id: UUID, name: string, icon: string, ...}[]>`

## Risks & Blockers

### Risk 1: JSON Files Incomplete or Missing (Medium)
- **Issue:** `/src/data/*.json` files may be missing, incomplete, or have unexpected structure
- **Mitigation:** Verify all 8 category files exist in `/src/data/` during Task 1.7.1 kickoff
- **Impact:** If files missing → seed fails → need to investigate file structure
- **Ownership:** @data-engineer checks before writing migrations

### Risk 2: Category Schema Mismatch (Low)
- **Issue:** Current JSON file structure may differ from planned table schema (points format, difficulty enum values, etc.)
- **Mitigation:** During Task 1.7.1, review one JSON file completely to confirm structure matches expectations
- **Impact:** Seed INSERT statements may need adjustment, +30 min rework
- **Ownership:** @data-engineer verifies schema before Task 1.7.2

### Blocker: RLS Policies Not Verified (High)
- **Issue:** Without RLS verification, public data exposure risk
- **Dependency:** STORY-001.011 (Verify RLS Policies) handles comprehensive testing
- **Resolution:** Cannot merge PR until RLS verified in Story 001.011
- **Ownership:** @qa gates this in final review

### Risk 3: Cache TTL Configuration (Low)
- **Issue:** 5-minute cache TTL may not be appropriate for all use cases
- **Mitigation:** Set to 5 min, but document how to adjust if needed
- **Impact:** Challenges shown to users may be up to 5 min stale
- **Ownership:** @dev implements, configurable via supabase-client.ts constant
